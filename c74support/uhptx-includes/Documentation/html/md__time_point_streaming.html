New in SDK version 2

The Time Point Streaming Emitter gives users direct control over the generated waveform, and the array can be updated at a much faster rate. This, combined with the user-supplied waveforms, unlocks a whole new range of possible sensations. To do this, the original Emitter class has been split into two distinct versions, an Ultrahaptics::AmplitudeModulation::Emitter which functions the same as the original Emitter object in our version 1 SDK and a new Ultrahaptics::TimePointStreaming::Emitter which exposes this new functionality.

NOTE
The Time Point Streaming API is more complex than the Amplitude Modulation API, and it does require some low-level code to use effectively. It may not be appropriate for use in simpler projects or projects which are already well established using Amplitude Modulation.
The Time Point Streaming API gives you direct control of the waveform's output by the device. This means that in order to generate a point which can be felt, you must apply a modulation to the intensity of the point. This is covered in more detail below.

How Time Point Streaming works

The Ultrahaptics::TimePointStreaming::Emitter is based upon a new callback mechanism. Instead of pushing updates to the device as in Amplitude Modulation, the user registers their own callback function with the emitter. This callback function will be called periodically by the Ultrahaptics API, requesting output control point data for the array for a given time period. These periods are joined together to give a continuous haptic output on the device.

Time Point Streaming in action

Time Point Streaming walkthrough

High level overview of using the Time Point Streaming API:

  1. Create a Time Point Streaming emitter
  2. Set the maximum control point count, i.e. the maximum number of control points you wish to project using this emitter with setMaximumControlPointCount
  3. Write an emission callback function
  4. Register the emission callback function with the emitter using setEmissionCallback
  5. Start the emitter
  6. Your callback function will be called every so often to request control point data
  7. When your callback is called, you are presented with a window of time (an output interval) to specify data for. Each point in time is one sample for the device. You'll need to iterate through these samples, updating the position etc. of all the persistent control points in the time window. You can find out the duration of one sample by calling getSamplingInterval
  8. When you want to stop outputting from the array, there are a few options: you can call stop on the emitter, or you can disable individual control points, or you can set the intensity of the control points to 0.
NOTE
Note that the duration of the OutputInterval may not correspond to a complete period of your chosen waveform. The length of the output interval is a period of time based upon the sample rate the device can achieve with your host PC, given the maximum number of control points and the latency communicating with the device. It is expected that waveforms will span multiple output intervals.

The callback in more detail

Let's take a look at how control points are updated using this new callback mechanism.

Broadly speaking, there are two ways to do this. You can either do it on-the-fly in your callback function, or you can precompute the point states in advance and iterate through them. The method you choose will depend upon your specific scenario. Upon installing the API you'll find a set of example programs using the new API. Two of these project a point 15cm above the centre of the array. Both use cosine modulation, and they achieve this using these two different ways.

  • TimePointStreaming_CachedPoint precomputes the positions of the control points through one complete 'cycle' of device updates. The callback function iterates through these point positions, looping around the list of precomputed points indefinitely.
  • TimePointStreaming_Focus calculates the control point intensity on-the-fly by defining a frequency to modulate at, and then generating a cosine wave based upon the time elapsed for one sample.

A closer look at TimePointStreaming_CachedPoint.cpp

Here's the callback function from the TimePointStreaming_CachedPoint example in more detail:

void my_emitter_callback(
const Ultrahaptics::TimePointStreaming::Emitter & timepoint_emitter,
Ultrahaptics::TimePointStreaming::OutputInterval & interval,
const Ultrahaptics::TimePoint & submission_deadline,
void * user_data_ptr) {
// The user pointer allows us to store data point we can re-use in our callback.
// In this case, we've used a custom struct that describes the precomputed control point
// data for the number of samples required to represent a complete period of our waveform.
UserData * user_data = static_cast<UserData *>(user_data_ptr);
// Iterate through all of the time points in the given output interval.
// We only have one control point, and at each point in time we are updating the control
// point to the next precomputed state.
// This outputs a continuous smooth waveform on the array.
for (Ultrahaptics::TimePointStreaming::OutputInterval::iterator
it = interval.begin();
it < interval.end();
++it) {
// Update the control point at this time point with our precomputed value
it->persistentControlPoint(0) = user_data->timepoints[user_data->next_index];
// Update our custom struct so we know which control point we are going to use for the
// next time point. Because we have precomputed one period of the waveform, we wrap
// around to the start of our precomputed vector of control points.
user_data->next_index = (user_data->next_index + 1) % user_data->timepoints.size();
}
}
NOTE
We'd recommend starting with a cosine modulation of your control points, as this tends to produce strong haptic effect with quiet operation of the array. Take a look at TimePointStreaming_CachedPoint in the examples folder for an example of cosine modulation.

Using custom modulation

With Time Point Streaming you can modulate the points in any way you choose, by varying the intensity of the control point at each point in time. This variation might be based upon a chosen modulation function, or you could use a stream of data to modulate the points. We have provided a source code example along with this SDK, TimePointStreaming_FreeformTracking, which loads a custom audio file and modulates a control point with the waveform which represents the sound. We've found that this can be a very intuitive and useful way to generate sensations, especially those with associated audio feedback.

If you would like to experiment with modulation based upon existing audio, this is quite simple using the free audio tool Audacity. To prepare a waveform which can be tested with the code in the TimePointStreaming_FreeformTracking example, you can follow these steps:

  1. Load your chosen audio file in audacity
  2. Click on 'Tracks' and then use 'Stereo Track to mono' to combine the left and right tracks
  3. Click on 'File' and then 'Export Audio'
  4. Use the file type 'Other uncompressed files' and choose 'WAV (Microsoft)' and 'Signed 32-bit PCM'
  5. If your chosen audio file has negative amplitudes (i.e. values between -1 and 0) you may want to account for these because negative intensity does not output anything on an Ultrahaptics control point. The easiest way to do this is to modify the example code to add 1 to the intensity values applied to the control point. Other strategies might involve shifting the waveform data so it never goes below zero and never above 1 (which would compress the range).
  6. Compile the SDK examples and then pass the path to this new WAV file as an argument to TimePointStreaming_FreeformTracking
  7. The example code will conform the audio to match the Ultrahaptics array's update sample rate, so you don't need to modify this yourself

Using more than two control points

Given the high update rate of Time Point Streaming, you should not usually require more than one or two control points. If you intend to use more than two control points with Time Point Streaming, and your device supports them, you may need to make use of control point groups.

This feature allows you to split your control points into separate groups, which are solved independently. Shifting the phase of the 200 Hz modulation signal of one group by 180° smooths out the power consumption so only half as many control points are at full modulation amplitude at a time. Control point groups work better if their control points are near each other.

You can set groups on Time Point Streaming control points via their setGroup method. This allows you to tell the SDK how it can group control points together, which helps produce a stronger haptic effect. For the optimum output strength, you should try to group points which are turned on and off at the same time (with your custom modulation) into the same group. Using more than two groups may result in degraded performance.

Currently only the UHEV1 supports control point groups.

Backwards compatibility

The V2 SDK includes an Amplitude Modulation emitter, which is backwards compatible with the emitter from version 1 of the SDK.

The V2 SDK includes a header file Ultrahaptics.hpp. This header file is present for backwards compatibility reasons, and it includes the AmplitudeModulation emitter. This means that code which has been written against V1 of the SDK will compile and run using the V2 SDK. It will use the Amplitude Modulation emitter and no code changes will be required.

The V2 SDK defines two additional header files, UltrahapticsAmplitudeModulation.hpp and UltrahapticsTimepointStreaming.hpp. It is intended that one of these header files will be used to explicitly define the type of emitter you want to use.
