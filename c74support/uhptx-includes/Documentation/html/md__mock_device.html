The Ultrahaptics SDK can now emulate a hardware device and create a log of the communications which would be sent to the device. This allows the SDK to be used without hardware, for testing or debugging purposes.

Creating a Mock Device

Mock Device is specified when creating an Emitter object, by using the device identifier MockDevice:, followed by the desired device type.

For example, the following code will create a AM emitter using a mock device which will behave as if an Ultrahaptics TOUCH Development Kit (UHDK5) were connected: 

Ultrahaptics::AmplitudeModulation::Emitter emitter("MockDevice:U5");

The mock device mode currently supports three device types - the Ultrahaptics Evaluation Kit (UHEV1) Dragonfly, and the Ultrahaptics TOUCH Development Kit (UHDK5) U5, and the Ultrahaptics STRATOS Explore Development Kit (USX) USX.

It may be useful to index a mock device if more than one is being used. This is possible using the serial option; append serial= to the end of your device identifier split by a semicolon. For example, using the device identifier MockDevice:U5;serial=1 will create a device with serial number MOCKMOCK-1.

It is also possible to specify the array layout for the UHDK5 by using the layout option, by default the device will imitate a square U5, however using the device identifier MockDevice:U5;layout=2 will create a mock device imitating a rectangular UHDK5.

Mock Device Logging

In order to log the device communications, a log file should be specified; this is achieved by appending the mock device identifier with the option logfile= followed by the desired name of the output file. Alternatively, leaving no file name will cause output to be sent to stdout.

For example, the device identifier MockDevice:U5;logfile=log.txt will create a file named log.txt in the working directory containing the output to the device.

Logging Format

By default the log file will contain tab separated values of the output sent to the device in the following formats:

For a UHDK5: cp# x y z intensity frequency time

For a USX: cp# x y z intensity time ticks

For a UHEV1: cp# x y z magnitude time

Where cp# refers to control point number (indexed at 0), ticks is an imitation of the device time which depends on the sample rate and magnitude is proportional to the intensity and time is the time of the update in seconds.

Note: magnitude is proportional to intensity for one control point, however, for multiple control points their values may fluctuate somewhat from direct proportionality to intensity.

There are also three alternative output format options, these may be specified by using the format option (e.g. MockDevice:U5;logfile=log.txt;format=1). Specifying format 0 will produce the outputs shown above. The other formats are as follows:

Format 1: cp# x y z

Format 2: cp# x y z intensity/magnitude

Format 3: cp# x y z time
